# ML 評估改進

**日期：** 2026-04-06  
**狀態：** ✅ 完成

---

## 🎯 我們修復的問題

之前，這個專案有一個**資料洩漏問題**：
- 只有 2 個分割：90% 訓練集，10% 驗證集
- 驗證集用於檢查點選擇
- 模型透過訓練過程「間接看到」驗證資料
- 報告的準確率有樂觀偏差

**這違反了基本的 ML 規範！**

---

## ✅ 所做的變更

### 1. 適當的訓練/驗證/測試分割

**之前：**
```python
# datasets.py - 錯誤
train_size = 90%
val_size = 10%
# 沒有測試集！
```

**之後：**
```python
# datasets.py - 正確
train_size = 80%  # 用於訓練
val_size = 10%    # 用於檢查點選擇（間接看到）
test_size = 10%   # 從未接觸 - 真實性能
```

### 2. 訓練中的測試集評估

**新增至 `train.py`：**
```

在從未在訓練期間看到的保留測試集上評估...

   損失：      0.0461
   詞元準確率： 98.67%
   序列準確率： 98.60%   ← 這是真實指標！
   困惑度：     1.05
```

**更新 `benchmark.py`：**
- 從訓練輸出解析測試集結果

- 將測試指標儲存至 JSON

**輸出：**
```

```

## ⚠️ 我們沒有新增的內容（刻意的）

刻意跳過這些**超出範圍**的功能：

❌ **多次運行與信賴區間**
- 範例：「98.2% ± 0.4%（n=5）」
- 為何跳過：超出範圍，此專案關於 Transformers 而非統計學

❌ **統計顯著性測試**
- 範例：t 檢定、p 值
- 為何跳過：對教育專案過度了

❌ **學習曲線圖**
- 範例：損失 vs epoch 圖表
- 為何跳過：不錯但會分散對架構的關注

❌ **混淆矩陣**
- 範例：每類準確率分解
- 為何跳過：不適用於序列任務

❌ **交叉驗證**
- 範例：5 折交叉驗證
- 為何跳過：計算過度

❌ **錯誤分析框架**
- 範例：分類失敗模式
- 為何跳過：需要單獨的專案

**理由：** 此專案是「從零開始的 Transformers」，而非「ML 評估最佳實踐」。我們修復了資料洩漏問題（必須的），但沒有將其變成統計教科書。
